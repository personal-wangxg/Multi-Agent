## 1. 设计目标

FR-17 可视化界面是 EduAgents 框架的 P2 可选功能，但必须预留 API 接口供未来扩展。界面核心目标是提升教师的教学规划效率，提供直观的任务状态可视化和便捷的人机协同操作。

| 设计原则 | 说明 |
|---------|------|
| **预留优先** | API 接口先行，UI 渐进实现 |
| **高效操作** | 关键操作一步直达，减少交互成本 |
| **状态透明** | Agent 执行状态、任务进度实时可见 |
| **人机协同** | 教师可随时介入、审批、修改 Agent 输出 |

## 2. 技术架构

### 2.1 技术栈选择

| 层级 | 技术选型 | 理由 |
|-----|---------|------|
| **前端框架** | React 18 + TypeScript | 生态成熟，组件复用率高 |
| **状态管理** | Zustand / Redux Toolkit | 轻量级状态管理 |
| **UI 组件库** | Ant Design Pro | 企业级组件，表单/表格/图表完备 |
| **图表可视化** | D3.js / ECharts | 知识网络、流程图可视化 |

| **实时通信** | WebSocket (Socket.IO) | 任务状态实时推送 |

### 3.6 人机协同操作界面

#### 3.6.1 五种操作支持

| 操作 | 说明 | UI 表现 |
|-----|------|--------|
| **approve** | 直接通过当前产物 | 绿色「通过」按钮 |
| **revise** | 教师直接修改后通过 | 打开编辑表单，确认后提交 |
| **reject** | 打回并说明原因 | 红色「打回」按钮 + 原因输入框 |
| **delay** | 延迟处理 | 黄色「稍后处理」按钮 |
| **approve_with_comments** | 通过但附加意见 | 蓝色「通过+备注」按钮 + 备注输入框 |

#### 3.6.2 反馈表单结构

```yaml
feedback_form:
  task_id: "task_001"
  stage: "stage2_structure"
  action: "revise"                    # approve / revise / reject / delay / approve_with_comments
  
  # action=revise 时填写
  revisions:
    - field: "units[1].periods"
      original_value: 3
      new_value: 5
      reason: "第2章内容较多，需要更多课时"
  
  # action=reject 时填写
  rejection_reason: "课时分配不符合课程目标"
  
  # action=approve_with_comments 时填写
  comments: "整体结构合理，但建议增加实践环节"
  
  # 通用
  annotation: "补充说明（可选）"
```

## 4. API 接口设计

### 4.1 API 位置预留

根据 DP-ARCH-06，FR-17 可视化界面是 P2 可选功能，但必须预留 API 接口位置。API 接口定义在以下路径：

```
eduagents/api/
├── __init__.py
├── v1/
│   ├── __init__.py
│   ├── tasks.py              # 任务管理 API
│   ├── networks.py           # 知识网络 API
│   ├── agents.py             # Agent 配置 API
│   ├── feedback.py           # 人机协同反馈 API
│   ├── observability.py      # 可观测性 API
│   └── config.py             # 配置管理 API
└── openapi.yaml              # OpenAPI 规范文件
```

### 4.2 核心 API 列表

| 路径 | 方法 | 功能 | 认证要求 |
|-----|------|------|---------|
| `/api/v1/tasks` | POST | 创建任务 | teacher |
| `/api/v1/tasks` | GET | 列出任务 | teacher |
| `/api/v1/tasks/{task_id}` | GET | 任务详情 | teacher |
| `/api/v1/tasks/{task_id}/stages/{stage}/feedback` | POST | 提交反馈 | teacher |
| `/api/v1/networks` | GET | 列出知识网络 | teacher |
| `/api/v1/networks/{network_id}` | GET | 网络详情 | teacher |
| `/api/v1/networks/{network_id}/visualization` | GET | 获取可视化数据 | teacher |
| `/api/v1/agents` | GET | 列出 Agent 配置 | admin |
| `/api/v1/agents/{agent_id}` | PUT | 更新 Agent 配置 | admin |
| `/api/v1/observability/logs` | GET | 查询执行日志 | admin |
| `/api/v1/observability/audit` | GET | 查询审计追踪 | admin |

### 4.3 统一响应格式

```yaml
api_response:
  success: true/false
  data: {}                          # 成功时返回的数据
  error:
    code: "ERROR_CODE"              # 错误码
    message: "错误描述"
    details: {}                     # 详细信息（可选）
  meta:
    request_id: "req_001"           # 请求追踪 ID
    timestamp: "2026-06-15T10:00:00Z"
    page: 1                         # 分页信息
    page_size: 20
```

### 4.4 错误码定义

| 错误码 | HTTP 状态码 | 说明 |
|-------|-------------|------|
| `TASK_NOT_FOUND` | 404 | 任务不存在 |
| `STAGE_NOT_FOUND` | 404 | 阶段不存在 |
| `INVALID_ACTION` | 400 | 无效的操作类型 |
| `UNAUTHORIZED` | 401 | 未认证 |
| `FORBIDDEN` | 403 | 无权限 |
| `CONFLICT` | 409 | 状态冲突（如重复提交） |
| `INTERNAL_ERROR` | 500 | 内部错误 |

## 5. 权限控制

### 5.1 角色定义

| 角色 | 权限范围 |
|-----|---------|
| **teacher** | 查看/审批自己创建的任务，提交反馈，查看知识网络 |
| **admin** | 所有 teacher 权限 + Agent 配置管理 + 系统配置 |
| **viewer** | 仅查看（只读），无操作权限 |

### 5.2 权限矩阵

| 操作 | teacher | admin | viewer |
|-----|---------|-------|--------|
| 创建任务 | ✓ | ✓ | ✗ |
| 查看任务 | 仅自己创建 | 所有 | ✓ |
| 审批任务 | 仅自己创建 | 所有 | ✗ |
| 配置 Agent | ✗ | ✓ | ✗ |
| 查看知识网络 | ✓ | ✓ | ✓ |
| 编辑知识网络 | 仅审批流程中 | ✓ | ✗ |
| 查看日志/审计 | ✗ | ✓ | ✗ |

## 6. 性能要求

### 6.1 响应时间要求

| 操作 | 响应时间要求 | 说明 |
|-----|-------------|------|
| 页面初始化 | ≤ 2s | 首屏加载时间 |

| 指标 | 要求 |
|-----|------|
| 同时在线教师数 | ≥ 50 |

## 7. 关键设计决策

| 决策点 | 设计选择 | 理由 |
|-------|---------|------|
| 前端框架 | React + TypeScript | 组件化程度高，类型安全 |
| 后端框架 | FastAPI | 异步高性能，自动 OpenAPI |
| 可视化库 | D3.js + ECharts | D3 灵活绑定，ECharts 图表丰富 |
| 实时通信 | WebSocket | 任务状态实时推送需求 |
| API 优先 | 先定义 API 再实现 UI | 保证接口标准化，UI 可独立迭代 |
