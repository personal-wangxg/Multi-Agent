# DP-ARCH-10：上下文窗口控制

**确认日期**：2026-06-12 · **确认人**：项目核心团队 · **决策类别**：架构

---

## 决策摘要

LLM 的上下文窗口是有限资源。系统采用**硬约束 + 分层控制**：单 Agent 输入 Token 必须≤ 模型窗口的 50%，决策索引摘要注入≤ 300 Token，全局上下文膨胀速率≤ 5%/轮。LLM 调用前必须计算当前输入 Token 量，超限执行压缩/截断/卸载三种策略。

## 核心设计原则

1. **硬约束优先：LLM 调用前必先执行 Token 计数检查，
2. **极简上下文主义：通过决策索引摘要而非完整产物注入上下文，
3. **主动压缩策略：超限 → 摘要压缩 / 旧消息截断 / 中间产物卸载，
4. **膨胀速率控制：每轮上下文增长不超过 5%，
5. **分层 Token 预算：STM/WM/LTM 各层级各有独立上限。

## 关键细节

### 硬约束数值

| 约束项 | 上限值 | 说明 |
|--------|--------|------|
| 单 Agent 输入 Token | ≤ 模型窗口 × 50% | LLM 调用前硬性检查，超限不发起 |
| 决策索引摘要注入 | ≤ 300 Token | context_injection_summary 固定上限 |
| WM 单文件 Token | ≤ 4096 Token | 中间产物文件上限，超限自动分片 |
| LTM 单次检索注入 | ≤ 1500 Token | 从向量检索上限 |
| 全局上下文膨胀速率 | ≤ 5%/轮 | 每轮对话上下文增长速率上限 |

### LLM 调用前 Token 检查流程

```
准备发起 LLM 调用
        │
        ▼
计算当前输入 Token 总数 =
(STM 消息 + 决策索引摘要 + LTM 检索注入)
        │
        ▼
   Token ≤ 模型窗口 × 50%？
        │
        ├── 是 → 正常发起 LLM 调用
        │
        └── 否 → 触发压缩策略
                  │
                  ├── 策略 A：摘要压缩（将旧消息转为 2-3 条合并为摘要）
                  │
                  ├── 策略 B：旧消息截断（丢弃最早的非关键消息
                  │
                  └── 策略 C：中间产物卸载（将中间产物写回 VFS，从上下文移除，仅保留索引）
```

### 超限三策略对比

| 策略 | 触发条件 | 操作 | Token 节省量 |
|-----|---------|------|------------|
| A（摘要压缩） | 超出 ≤ 20% | 将 2-3 条旧对话合并为 1 条摘要 | 约 30-50% |
| B（旧消息截断） | 超出 20-50% | 丢弃最早的非关键历史消息 | 约 40-60% |
| C（中间产物卸载） | 超出 ≥ 50% | 将完整中间产物写回 VFS，上下文仅保留决策索引 | 约 70-90% |

### 上下文膨胀速率控制机制

每轮对话后检查：`(本轮上下文 Token - 上轮上下文 Token) / 上轮上下文 Token ≤ 5%

若 > 5%，自动触发策略 A 摘要压缩。

### 与三层记忆体系的配合

```
STM（当前上下文）：
  └→ 决策索引摘要（≤ 300 Token）
       └→ WM 中完整产物（Agent 4096 Token/文件，主动读取）
            └→ LTM 中历史模板（≤ 1500 Token/检索）
```
