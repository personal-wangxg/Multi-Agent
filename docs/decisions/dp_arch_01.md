# DP-ARCH-01：采纳方案D — 知识编译 + 决策索引

**确认日期**：2026-06-12 · **确认人**：项目核心团队
**决策类别**：架构 · **优先级**：P0（核心架构）

---

## 决策摘要

在多阶段/多场景的长链路 Agent 协作中，每个阶段的完整产物**不直接拼入后续阶段上下文**。取而代之，由独立"编译 Agent"将每个阶段产物自动编译为结构化决策索引（Decision Point 表），同时生成 Knowledge Wiki 页面。后续阶段以极简决策索引注入上下文，如需详细信息由 Agent 主动通过工具读取。

## 核心设计原则

1. **决策前置编译，而非查询时检索**
   - 教师确认一个阶段的产物后，立即进行知识编译提取 DP 表
   - 后续阶段直接消费已编译结果，不触发重新检索

2. **决策点不可变更**
   - 每个 DP 一经教师确认即为不可变
   - 修改仅通过"迭代新方案"方式生成新 DP，旧 DP 保留历史版本

3. **上下文极简主义**
   - 后续阶段上下文仅注入决策索引表（预计 ≤ 300 Token）
   - 不注入完整产物全文

4. **主动读取替代被动注入**
   - Agent 如需查阅决策点对应的详细上下文/推导过程
   - 通过 read_wiki(path) 或 read_yaml(path) 工具主动获取

5. **Git 版本管理**
   - Knowledge Wiki 和完整产物文件均由 Git 仓库管理
   - 支持 diff 对比、回滚、分支

6. **不需要向量数据库**
   - 知识已编译为结构化页面和索引，查询时直接读取即可

## 与其他方案的对比

| 方案 | 信息完整性 | 上下文可控 | 可追踪性 | 实现复杂度 | 决策 |
|-----|-----------|-----------|---------|-----------|------|
| 方案A（纯 RAG） | ⚠️ 依赖检索命中率 | ✅ 可控 | ❌ 不可控 | 高（向量DB） | 不采纳 |
| 方案B（摘要+引用） | ✅ 完整产物写入文件 | ✅ 摘要固定长度 | ✅ 可审查摘要内容 | 低 | 基础版本 |
| **方案D**（Wiki编译+决策索引） | ✅ 完整产物+结构化索引 | ✅ 300 Token 索引 | ✅ 每个DP有ID，精确追踪 | 中 | **采纳** |
